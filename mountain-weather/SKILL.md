# 山の天気の判断

登山対象の山の天気を、登山指数・麓の気象・標高差からの推定で総合的に判断する手順。

## 前提: 何が取れて何が取れないか

山頂のピンポイントな気象数値そのものは一般に保証されないが、判断材料となる指標は無料で取得できる。

| ソース                              | 取得方法                         | 中身                                                               | 山頂相当か                              |
| ----------------------------------- | -------------------------------- | ------------------------------------------------------------------ | --------------------------------------- |
| てんくら (`tenkura.n-kishou.co.jp`) | curl + iconv (無料のサーバ HTML) | 登山指数 A/B/C、約 1 週間分、上空の気温・風、天気アイコン          | 山ごとの独自指数 (山頂を含む予報ベース) |
| 気象庁 (JMA)                        | curl + jq (無料の JSON)          | 麓・登山口を含む細分区の天気・風・降水確率、麓のアメダス地点の気温 | 麓・地域レベル                          |

判断の主軸はてんくらの登山指数とし、気象庁で麓の実データを併せて見る。

## 限界をユーザに伝える

- てんくらの登山指数 (A/B/C) は同社が独自基準で算出した指標であり、山頂の実況気象そのものではない。気象庁データは麓・地域レベルで、山頂とは乖離しうる。
- 山は標高・地形で天気が急変する。麓が晴れでも山頂は強風・荒天ということが普通に起きる。指数や麓の予報を山頂の確定予報のように断定して伝えない。
- 標高差から算出する山頂気温などは推定であると明記する。判断に迷う場合は危険側 (寒い・荒れる前提) に倒し、下振れ (暖かい・穏やか) に丸めて軽装・強行を促さない。

## DANGER: 値がズレると危険なので慎重側に倒す

山の気象判断は安全に直結する。指数・気温・天気を実際より楽観的に (良い指数・暖かい・穏やか) 報告した結果、ユーザが軽装・強行で入山して事故った場合の被害は甚大である。次のガードレールを必ず守る。

- 下振れ方向への丸めは禁止する。不確実な値は危険側 (悪い指数・寒い・強風・荒天前提) に倒す。良い方向への切り上げ・楽観はしない。
- 推測で値を埋めない。取得できない・確度が低いときはその旨を明示し、安全側で伝える。
- 複数ソース (てんくら・気象庁・推定) が食い違うときは楽観側を採らない。悪い方の見立てを主に置き、両方を併記する。
- 山頂気温は逓減率と風の影響で寒い側に寄せる (後述「山頂気温の概算」と接続)。氷点下・強風・発雷・荒天の兆候は省略せず必ず強調する。
- 最終判断はユーザに委ねる。skill 側で「行ける」「問題ない」と断定しない。中止・撤退・日程変更の選択肢を常に残す。

## 1. てんくらの登山指数 (主軸)

てんくらは旧来のサーバ描画 HTML (`/tk/kanko/`) を残しており、ここから登山指数 (A/B/C) を無料で取得できる。文字コードは Shift_JIS なので、 iconv で UTF-8 に変換してから抽出する (変換しないと grep が日本語を拾えない)。

### 山のコードを引く

地域ごとの山リスト `kasel.html` から、対象の山の名前に対応するコード (`kad.html?code=` の数値) を引く。

```sh
# 地域 (ba) と登山タイプ (type=15) を指定して山リストを取得
curl -sSL -A "Mozilla/5.0" \
  "https://tenkura.n-kishou.co.jp/tk/kanko/kasel.html?ba=hk&type=15" \
  -o /tmp/tenkura-kasel.html
iconv -f SHIFT_JIS -t UTF-8//IGNORE /tmp/tenkura-kasel.html > /tmp/tenkura-kasel.utf8.html

# 山名とコードの対応を抽出 (例: kad.html?code=01150271...>青山)
grep -oE 'kad\.html\?code=[0-9]+[^<]*>[^<]+' /tmp/tenkura-kasel.utf8.html
```

地域コード `ba` は次のとおり。コードの数値の先頭 2 桁は都道府県コード (例: `01`= 北海道) と対応する。

```
hk 北海道    th 東北    kk 関東・甲信    hr 北陸    tk 東海
kn 近畿    cg 中国    sk 四国    ks 九州・沖縄
```

- 同名の山が複数あることがある (例: 別の山域の同名峰)。コードの先頭 2 桁の都道府県や標高で対象を確定する。一意に定まらなければユーザに確認する。憶測で 1 つに決めない。候補をユーザに提示するときは、都道府県 (コード先頭 2 桁) ・標高 ・ `code` を識別キーとして列挙し、通称への変換は推測しない。
- リストには山頂以外の地点も混じる (`赤岳鉱泉` `赤岳山荘` など)。名前が `山荘` ・ `小屋` ・ `鉱泉` などの施設語で終わる項目は山頂候補と分けて扱う。
- `type=15` が登山。他の値は観光・桜などの別カテゴリなので、登山指数を取るなら `type=15` を使う。
- 県境・独立峰の山はてんくらのエントリが登山口と別の県・別の `ba` に入ることがある (例: 富士山は `ba=kk` ・コード先頭 19 = 山梨だが登山口は静岡)。 `ba` は省略でき `code` + `type=15` で直に引けるので、コードが分かれば `ba` に依存しない。
- リスト (kasel) 取得も CDN キャッシュで別地域が返ることがある。取得直後に `<title>` の地域名やコード先頭 2 桁の分布が要求地域と合うか確認し、違えばキャッシュを回避して取り直す (検証の詳細は「抽出の検証」)。

### 登山指数を取得する

山のコードで個別ページを取得する。`ba` は省略してよく、`code` と `type=15` だけで登山指数が返る。

```sh
curl -sSL -A "Mozilla/5.0" \
  "https://tenkura.n-kishou.co.jp/tk/kanko/kad.html?code=01150271&type=15" \
  -o /tmp/tenkura-kad.html
iconv -f SHIFT_JIS -t UTF-8//IGNORE /tmp/tenkura-kad.html > /tmp/tenkura-kad.utf8.html
```

見出し `登山指数と上空の気温・風` の領域は、1 日 = 1 つの `<table>` で並ぶ。表の行はクラスで意味が決まり、列は左から時間帯ヘッダと対応する (指数・気温・風は同じ列が同じ時間帯を指す)。grep で値だけ拾うと列対応を取り違えるので、行クラスで構造を掴んでから列位置で対応付ける。

| 行            | 行クラス・目印                                  | 中身                                                                                                                                                    |
| ------------- | ----------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 日付          | `tr.tit_day`                                    | `今&nbsp;日&nbsp;6/28(日)` など                                                                                                                         |
| 時間帯ヘッダ  | `tr.t_b`                                        | `00 03 06 09 12 15 18 21` (今日・明日は 3 時間ごと 8 列)                                                                                                |
| 登山指数      | `tr.t_w.mnt_td`                                 | 各列の `<img alt="登山指数">`                                                                                                                           |
| 気温 (気圧面) | `tr.t_w` (先頭セルに `NNNhPa` が `rowspan="2"`) | 各列の気温セルは `<td id="temp-d"><b>N℃</b>`                                                                                                            |
| 風 (気圧面)   | 上の直後の `tr.t_w`                             | 風向は `<td id="wind-d">` 内の `wind_*.gif`、風速は `<span class="kwinNN">N</span>` のテキスト `N` (m/s)。 class の `NN` は色分けなので速度と混同しない |

- 登山指数は `mnt_td` 行の画像ファイル名で表す。 `mnt1.gif`= A、 `mnt2.gif`= B、 `mnt3.gif`= C。 A が登山に適、 C が不適。
- 気圧面 (`NNNhPa`) は山の標高付近が選ばれる (低山は 925/950hPa、高山は 600/700hPa)。高度の目安は 950hPa が約 500m、925hPa が約 760m、850hPa が約 1500m、800hPa が約 2000m、700hPa が約 3000m、600hPa が約 4400m。山頂相当の気温・風は次の順で決める。 (1) 山頂標高に最も近い気圧面を使う。 (2) 山頂が 2 面のほぼ中間でどちらにも近いときは 2 面を内挿する。 (3) 表示中でいちばん高い気圧面よりさらに上に山頂があるなら (例: 700hPa しか無く山頂がそれより上)、その面を基点に「3. 山頂気温の概算」の逓減率で山頂まで再外挿する。いずれも確証が持てなければ寒い側 (高い気圧面 = 低い気温) に倒す。
- 時間粒度は部位で変わる。今日・明日は 3 時間ごとの時間帯別、3 日目以降の週間部は当日 9 時予想の単一値 (時間帯別ではない)。週間部を時間帯別と取り違えない。
- 今日・明日の日付は `tr.tit_day` の見出しから採る。ただし週間部では `tr.tit_day` は「週間予報」の区分ラベルにすぎず、各列の実日付 (`6/30` `7/1` など) は `tr.t_b` (日付行) の列ヘッダに入る。週間部の日付は `t_b` から採る。本文の散文中に出る無関係な日付 (開山時期の定型文など) は判断に使わない。
- 時間帯別テーブルの先頭が常に「今日」とは限らない。日が進んで夜遅くになると今日分のテーブルが落ち、先頭が「明日」になることがある (時間帯 8 列は今日・明日のどちらも持つので、列数だけでは今日と断定できない)。どのテーブルがどの日かは必ず `tr.tit_day` の日付見出しで確定し、先頭 = 今日と決め打ちしない。
- 指数行 (`mnt_td`) は直前の `tr.tit_day` に束ねる。今日・明日は時間帯 8 列、週間部は 1 日 1 列で複数日が 1 行に並ぶ。ページはネストした `<table>` を含むので、`<table>` 単位で機械分割すると入れ子で日付境界が崩れる。 `tr.tit_day` の見出しと列数 (8 列 = 時間帯別/それ未満 = 週間) で日付を確定する。同じ `tit_day` が複数テーブル (8 列版と分割版) に重複出現することがある。列数が最大のものを正本とし、分割版は同一データの再掲として無視する。
- 表示される気圧面は日・標高条件で変わる (同じ山でも今日が 600/700hPa、明日が 700/800hPa などになる)。気圧面はハードコードせず、各テーブルの `NNNhPa` 表記を都度読んで山頂標高との関係を判定する。
- 指数・気温・風の各行は同じ日付 (時間帯) 列インデックスで対応する。各行の先頭セルはラベル列 (気温・風行は `NNNhPa`、週間の指数行は `登山指数` で `colspan="2"`) なので、その分ずらして値列を対応づける。週間部でも同じ。
- DANGER: 日付と指数の対応に `grep -A`/`-B`/`-C` などの行近接 (テキスト上の前後行) を使わない。日付行 (`tr.t_b`) と指数行 (`tr.t_w.mnt_td`) は別々の `<tr>` で、週間部では日付行に `7/1`〜`7/6` が縦に並び、その直後に指数行が続く。`grep -A5 '7/3'` は「7/3 の続きの日付 (7/4・7/5・7/6) → 指数行の先頭セル」を拾い、指数行の**先頭列 (= 7/1 の指数)**を 7/3 の指数と取り違える。対応は必ず列インデックスで取る (日付行の N 列目と指数行の N 列目が同じ日)。
- 週間部は指数 gif の数が日付列数より少ないことがある。最も先の日付は指数がまだ出ておらず、行末に gif の無いセル (`-`) が付く、または列自体が欠けることがある。指数は最も近い日付 (左) から順に日付列へ対応づけ (左詰め)、足りない分は最も先の日付を「指数なし (予報範囲外)」とする。末尾を残す (右詰め) と、近い日付の指数が 1 つずれて別日の値を読む。`<img>` だけを数えず `<td>` セル単位で扱い、先頭のラベルセルを除いてから左詰めで突き合わせる。
- 値と時間帯の対応に確証が持てないときは、指数 (A/B/C) を主に使い、気温・風の数値の断定は避け、山頂気温は「3. 山頂気温の概算」の逓減率で出して安全側に倒す。

```sh
# 指数・天気アイコンの出現を確認
grep -oE 'alt="登山指数"|mnt[123]\.gif|alt="天気"|/tenkim/[a-z][0-9]+\.gif' /tmp/tenkura-kad.utf8.html | sort | uniq -c
```

週間部の登山指数は、行近接ではなく列位置で日付に突き合わせる。日付行 (`t_b`) の直後の指数行 (`mnt_td`) を取り、先頭ラベルセルを除いて列を 1:1 で対応づける。`<table>` 分割版 (only-sm の 3 日ずつ) も同じ正規表現に当たるので、列数が最大の組 (only-pc の 6 日 1 行) を正本にする。この対応づけは `scripts/cli/weekly-index.ts` に実装済みで、iconv 済みの UTF-8 HTML を渡すと `日付 -> 指数` を出力する。手で grep せずこれを使う。

```sh
deno task --cwd .claude/skills/mountain-weather/scripts weekly-index /tmp/tenkura-kad.utf8.html
# => 7/1(水) -> A / 7/2(木) -> B / ... 欠測は "-"。詳細は scripts/README.md
```

### 抽出の検証

- `alt="登山指数"` のマッチが 0 件のときは、 `type=15` を付け忘れたか HTML 構造が変わった可能性が高い。値を捏造せず処理を止めて確認する。 `type` 無し・ `code` のみだと別カテゴリ (レジャー) のページが返り登山指数が無いので、必ず `type=15` を付ける。
- 指数 (`mnt[123]`) と日付・時間帯の対応がずれると別の日の指数を読む。日付見出しとの並びを確認し、ずれを疑ったらユーザに提示して確認する。
- 外部取得は CDN キャッシュで別の対象 (別の山) が返ることがある。これはてんくら・気象庁の両方で起きる。取得物が要求した対象か必ず検証する。てんくらは `code=` と `<title>`・近隣市町村名、気象庁は `publishingOffice` と `areas[].code` の先頭桁で照合する。違えばキャッシュを回避して取り直す (クエリ付与 `?_=1` か `-H 'Cache-Control: no-cache'`)。別の山・別県のデータを山頂判断に使うと危険。 CDN が不安定で同じ URL でも取得ごとに別の山が返ることがあるため、1 度の検証通過で安心せず、抽出の直前にも `code`/`title` を再照合する。検証と抽出は同一の cache-bust 済みファイルに対して連続して行い、間に別の取得を挟まない。
- iconv 変換後、日本語 (日付・山名等) が grep で拾えるか 1 度確認する。拾えない場合は元の文字コードが Shift_JIS でない可能性がある。
- 参考実装: <https://github.com/hidenorly/tenkura> (同じ `kad.html` を解析。 `mnt1/2/3.gif`→`A/B/C` の対応もここに準拠)。 HTML 構造が変わって抽出に失敗したらこの skill を更新する。

## 2. 気象庁の麓の天気 (補完・クロスチェック)

てんくらの指数に加え、麓・登山口周辺の実データ (天気・気温・降水確率) を気象庁から取る。気象庁の予報は気象台 (都道府県) 単位で、その中の細分区 (北部・南部・山沿いなど) に分かれる。

### 区域コードを引く (area.json)

```sh
curl -sS 'https://www.jma.go.jp/bosai/common/const/area.json' -o /tmp/jma-area.json

# 都道府県名 (正式名、例: 長野県) から気象台コード (office、6 桁) を引く
jq -r '.offices | to_entries[] | select(.value.name == "長野県") | .key' /tmp/jma-area.json

# その office 配下の細分区 (class10) 一覧
jq -r '.class10s | to_entries[] | select(.value.parent == "200000") | "\(.key) \(.value.name)"' /tmp/jma-area.json
```

どの細分区かは地理勘で決めず、登山口の市町村名から決定的に引く。 `area.json` の `class20s` (市町村・地区レベル) から `class15s` を経て `class10` (細分区) へ parent を辿る。

```sh
# 登山口の市町村・地区名 (例: 富士宮市、乗鞍上高地) から所属する細分区 (class10) を引く
jq -r --arg mun "富士宮市" '
  (.class20s | to_entries[] | select(.value.name | test($mun)) | .value.parent) as $c15
  | .class15s[$c15].parent as $c10
  | "\($c10) \(.class10s[$c10].name)"
' /tmp/jma-area.json | head -1
# => 220030 東部
```

- 細分区名は都道府県ごとに異なる (北部・南部、内陸・沿岸、南部山沿い・北部平野部など)。固定の名前を決め打ちせず、必ず上の方法で登山口の市町村から細分区を引く。
- `class20s` の地区名は細かい (例: `松本市乗鞍上高地`)。登山口に対応する地区語で `test` すると精度が上がる。市町村名でヒットしないときは、より広い市名や近隣地名で引き直す。
- てんくらが表示する「近隣市町村」は山頂位置基準で選ばれ、登山口と一致しないことがある。気象庁の細分区・アメダスは必ず登山口の実所在地から独立に引き、ソース間で地点が違えば回答に明記する。
- 山沿い・内陸の区分がある県では、山に近い区を優先する。平野部の天気は山と乖離しやすい。
- 北海道は気象台が複数に分かれるので、登山口を管轄する office を選ぶ。

### 予報を取得する (forecast)

```sh
curl -sS 'https://www.jma.go.jp/bosai/forecast/data/forecast/200000.json' -o /tmp/jma-forecast.json
```

配列は 2 要素で `[0]` が 3 日予報、`[1]` が週間予報。`[0].timeSeries` は 3 本あり区分単位が異なる。

| timeSeries | 内容                                                     | 区分単位                    |
| ---------- | -------------------------------------------------------- | --------------------------- |
| `[0]`      | 天気 `weathers` ・天気コード `weatherCodes` ・風 `winds` | 細分区 (class10)            |
| `[1]`      | 降水確率 `pops`                                          | 細分区 (class10)            |
| `[2]`      | 気温 `temps` (最低・最高)                                | アメダス観測地点 (麓の地点) |

天気・降水確率は細分区単位だが、気温だけはアメダス地点単位で粒度が違う。気温を細分区から探すと空振りする。

```sh
# 天気・風 (細分区コードで絞る。例: 中部 200020)
jq '.[0].timeSeries[0].areas[] | select(.area.code == "200020") | {area: .area.name, weathers, winds}' /tmp/jma-forecast.json
# 降水確率
jq '.[0].timeSeries[1].areas[] | select(.area.code == "200020") | {area: .area.name, pops}' /tmp/jma-forecast.json
# 気温 (アメダス地点。登山口に最も近い地点を名前で選ぶ)
jq '.[0].timeSeries[2].areas[] | {name: .area.name, temps}' /tmp/jma-forecast.json
```

- 気温のアメダス地点は登山口に最も近いものを名前で選ぶ。判断できない場合は複数地点を併記して幅で伝える。憶測で 1 地点に決め打ちしない。
- 各値がどの日時に対応するかは同じ `timeSeries` の `timeDefines` で必ず確認する。配列の index に意味を決め打ちしない (取得時刻によって入る要素が変わる)。
- 取得した forecast が別の県の内容になっていないか検証する。レスポンスの `publishingOffice` と `areas[].code` の先頭桁が要求した office と一致するか確認し、不一致なら (キャッシュ等で別府県を掴んでいる) 取り直す。別県の予報を山頂判断に使うと危険。取り直すときはクエリ文字列 (例: `?_=1`) を付けるか `-H 'Cache-Control: no-cache'` で CDN キャッシュを回避する。
- 数日先 (週間予報 `[1]`) は粒度が落ちる。 weather・降水確率が細分区ではなく office (県) 単位、気温がアメダス少数地点になり、登山口近傍の細分区・地点が得られないことがある。その場合は県単位・遠い地点の値だと明記して参考扱いにする。
- `publishingOffice`・`area.code` が正しい県でも、気温 `timeSeries[2]` のアメダスだけ別県が混入することがある (office・code 照合では気温の県跨ぎを検出できない)。気温地点名が対象県の地点か照合し、県外の地名が出たらキャッシュを回避して取り直す。別県の気温で山頂を概算するのは危険。
- 天気コードの和名は別ファイルに頼らず、同じ forecast の `[0].timeSeries[0].weathers` (日本語文字列) で同コードの意味を確認する。週間部だけに出るコードが `[0]` に無いときは、コードの百の位 (2xx = くもり系・3xx = 雨系・4xx = 雪系) で安全側 (降水・降雪あり前提) に解し、ピンポイントの和名は断定しない。

## 3. 山頂気温の概算 (標高差からの推定)

てんくらが上空の気温を出すが、取れない場合や麓のデータしかない場合は標高差から概算する。いずれも推定であり山頂の実測ではない。危険側に倒し、下振れに丸めない。なお、てんくらの上空気温は参照気圧面の値 (例: 800hPa は約 2000m) で、山頂標高より低いことがある。その場合はてんくら上空気温を基点に、下式で山頂標高まで再外挿する。

標高が 100 m 上がるごとに気温は約 0.6 度下がる (気温逓減率)。

```text
山頂気温の概算 = 麓の気温 - 0.6 × ((山頂標高 - 登山口標高) / 100)
```

- 逓減率 0.6 度/100m は標準的な目安。晴れて乾いた日は約 1.0、湿って雲が多い日は約 0.5 と幅がある。1 点の値で断定せず、迷うときは寒い側 (大きい逓減率) で見積もる。
- 体感温度は風速 1 m/s ごとに約 1 度下がる。稜線・山頂は麓より強く吹くのが通常なので、麓基準より厳しくなる前提で伝える。
- 最低気温側で氷点下・一桁になるなら、防寒・低体温のリスクとして必ず伝える。

## 注意

- 取得結果には対象の山・都道府県・細分区・アメダス地点・日時を必ず添える。どの地点・どの時間の値かを曖昧にしない。
- 登山指数は同社の独自指標、山頂気温・体感温度は推定であり、いずれも山頂の実況を保証しない。断定せず、安全側に倒して伝える。
- これらは明示的なオープンデータライセンスが確認できていない。取得した生データの無断の二次配布・公開はしない。
- weathernews 山の天気 (`https://weathernews.jp/mountain/`) も山ごとの指数を出すが、 JavaScript 描画の SPA で `curl` では取得できない。必要ならユーザにブラウザでの確認を案内するだけにとどめる。
- 一時ファイルは絶対パスで指定し (例: `-o /tmp/tenkura-kad.html`)、後続の `iconv` ・ `jq` も同じパスを参照する。
- 取得・抽出に失敗した場合 (`area.json` で都道府県が引けない/`kad.html` で登山指数が 0 件/forecast の構造が変わる等) は値を捏造しない。処理を止めて報告し、この skill を更新する。
