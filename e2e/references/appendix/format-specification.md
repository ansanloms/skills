# 試験項目書のフォーマット

## 概要

E2E テスト試験項目書の具体的な記述フォーマットを規定する。

## 基本記述形式

### タイトル(必須)

```markdown
# テスト項目の名称
```

### テスト概要(任意)

```markdown
このテストの目的、対象機能、前提条件等を記述。
第三者が理解できるよう背景情報を提供する。
```

### 共通テストケーステーブル(任意)

```markdown
| case | param1 | param2 | param3 |
| ---- | ------ | ------ | ------ |
| 1    | value1 | value2 | value3 |
| 2    | value4 | value5 | value6 |
```

### テストシナリオ(必須)

```markdown
## 1. シナリオ名

シナリオの概要や検証内容(任意)。

1. テストステップ1
2. @login-frontend
3. テストステップ3
```

- ステップに `@macro-name` と記述すると、`scenarios` 出力時にマクロ定義の値に展開される

### シナリオ独自テストケーステーブル(任意)

```markdown
## 1. シナリオ名

| case | param1 | param2 |
| ---- | ------ | ------ |
| 1    | valueA | valueB |
| 2    | valueC | valueD |

1. テストステップ1
2. テストステップ2
```

### デバイスプリセット定義(任意)

spec.md の先頭に YAML フロントマターとして記述する。

```yaml
---
devices:
  pc:
    viewport:
      width: 1920
      height: 1080
  mobile:
    viewport:
      width: 390
      height: 844
    isMobile: true
    hasTouch: true
    userAgent: Mozilla/5.0 (iPhone; CPU iPhone OS 16_0)
---
# テスト名
...
```

- キーはデバイス名（`pc`, `mobile` など）
- `viewport.width`/`viewport.height`: viewport サイズ（数値）
- `isMobile`/`hasTouch`: 真偽値（`true` または省略）
- `userAgent`: 文字列。`${ENV_VAR}` 置換記法を使用可

## パラメータ置換の記述

### 環境変数の記述

```markdown
${FRONTEND_URL} → 実行時に環境変数の値に置換
```

- 形式: `${変数名}`
- 用途: URL、認証情報等の環境依存値
- 例: `${FRONTEND_URL}/login`, `${TEST_USERNAME}`

### テーブル値の記述

```markdown
<param1> → テストケーステーブルの該当列の値に置換
```

- 形式: `<列名>`
- 用途: テストケース毎の入力値や期待値
- 例: `<email>`, `<expected_error>`, `<product_id>`

### マクロ参照の記述

```markdown
@login-frontend → 外部マクロファイルの login-frontend の script 値に展開（scenarios 出力時）
```

- 形式: `@マクロ名`（英数字・ハイフンのみ）
- 用途: よく使うサブテスト呼び出しコマンドのショートカット
- 例: `@login-frontend`, `@login-admin`
- マクロ定義は spec.json ではなく外部の macro.json で管理し、`/e2e run --macro <path>` で渡す
- 未定義のマクロ名はそのまま出力される（エラーにはならない）

## 記述例

### 例1: 共通テーブルを使用した基本例

```markdown
# ログイン機能のテスト

A!-IDを使用したログイン機能の動作確認。
正常ログインと異常ログインの両方をテストする。

| case | lang |
| ---- | ---- |
| 1    |      |
| 2    | /en  |

## 1. 正常ログインテスト

有効な認証情報でのログイン成功を確認する。

1. `${FRONTEND_URL}<lang>/login` にアクセスする
2. メールアドレス欄に `${TEST_USERNAME}` を入力する
3. パスワード欄に `${TEST_PASSWORD}` を入力する
4. ログインボタンを押下する
5. `${FRONTEND_URL}<lang>/` にリダイレクトされることを確認する

## 2. 異常ログインテスト

不正な認証情報でのログイン拒否を確認する。

1. `${FRONTEND_URL}<lang>/login` にアクセスする
2. メールアドレス欄に `invalid@example.com` を入力する
3. パスワード欄に `wrongpassword` を入力する
4. ログインボタンを押下する
5. エラーメッセージが表示されることを確認する
```

**この例の特徴**:

- 共通テーブル `lang` が両シナリオで使用される
- 環境変数 `${FRONTEND_URL}` 等が実行時に置換される
- 各シナリオがテーブルの行数分(2 回)実行される

### 例2: シナリオ独自テーブルを使用した例

```markdown
# フォームバリデーションテスト

入力フォームのバリデーション機能を確認する。

## 1. メールアドレス形式チェック

メール入力欄での形式バリデーションを確認する。

| case | email         | expected-error                       |
| ---- | ------------- | ------------------------------------ |
| 1    | invalid-email | メールアドレス形式が正しくありません |
| 2    | test@         | ドメインが不完全です                 |
| 3    |               | メールアドレスを入力してください     |

1. お問い合わせフォームにアクセスする
2. メールアドレス欄に `<email>` を入力する
3. 送信ボタンを押下する
4. エラーメッセージ `<expected-error>` が表示されることを確認する

## 2. パスワード強度チェック

パスワード入力での強度バリデーションを確認する。

| case | password | expected-result    |
| ---- | -------- | ------------------ |
| 1    | abc      | 文字数不足エラー   |
| 2    | abcdefgh | 文字種類不足エラー |
| 3    | Abc123!@ | バリデーション成功 |

1. 新規登録フォームにアクセスする
2. パスワード欄に `<password>` を入力する
3. 確認ボタンを押下する
4. `<expected-result>` の結果になることを確認する
```

**この例の特徴**:

- 各シナリオが独自のテーブルを持つ
- テーブル値 `<email>`, `<expected_error>` 等が各行の値に置換される
- シナリオ毎に異なる項目数でテストが実行される

### 例3: 単一ケースの例

```markdown
# 基本的なページ表示テスト

各ページが正常に表示されることを確認する。

## 1. トップページ表示テスト

トップページの基本的な表示を確認する。

1. `${FRONTEND_URL}/` にアクセスする
2. ページタイトルが「A!SMART」であることを確認する
3. ナビゲーションメニューが表示されていることを確認する
4. フッター情報が表示されていることを確認する

## 2. 会社概要ページ表示テスト

会社概要ページの表示を確認する。

1. `${FRONTEND_URL}/about` にアクセスする
2. ページタイトルが「会社概要」であることを確認する
3. 会社情報が表示されていることを確認する
```

**この例の特徴**:

- テストケーステーブルを使用しない単一ケース
- 各シナリオは 1 回ずつ実行される
- case 項番は常に 1 になる(1.1.1/1.1.2/1.1.3/1.1.4/2.1.1 ...)

## 記述時の注意点

### 必須要素

- タイトル(`# テスト名`)は必須
- 最低 1 つのシナリオ(`## シナリオ名`)は必須
- 各シナリオに最低 1 つのステップは必須

### 推奨事項

- テスト概要で目的と対象機能を明記
- 入力値や期待値が複数パターンあるテストケースはテーブルで整理する
- パラメータ名はわかりやすい名前を使用
- ステップは具体的で実行可能な内容を記述

### 制限事項

- テーブルヘッダーの列名は重複不可
- 環境変数名に使用可能な文字は英数字とアンダースコア
- テーブル列名に使用可能な文字は英数字とハイフン（アンダースコアは Markdown のエスケープ `\_` が発生するため非推奨）
- ステップ内では複数行にわたる複雑な操作は避け、可能な限り 1 行で表現

### テーブル設計指針

**共通テーブルを使用する場合**:

- 言語設定、環境設定等の共通パラメータ
- 全シナリオで同じデータパターンを使用する場合

**シナリオ独自テーブルを使用する場合**:

- バリデーションテスト等の入力値バリエーション
- エラーパターンテスト
- そのシナリオでのみ必要なデータパターン

**単一ケースを使用する場合**:

- 基本的な画面表示テスト
- 設定や状態に依存しない固定的なテスト
- データパターンが不要な単純なテスト
